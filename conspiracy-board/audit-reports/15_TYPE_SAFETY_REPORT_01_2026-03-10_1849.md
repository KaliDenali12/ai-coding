## 3. Type Safety Improvements Recommended (Not Implemented)

### Already Excellent — Minimal Recommendations

The codebase has **zero** explicit `any` types in production code, strict tsconfig enforced, and all type assertions are justified by runtime validation. This is above-average type safety.

**Minor structural improvements (low priority):**

- **Branded types for concept inputs**: `conceptA` and `conceptB` are plain strings throughout. A branded type like `ValidatedConcept` could prevent passing unvalidated strings to the API layer. However, given the small codebase size, this adds complexity without meaningful safety gain.

- **Discriminated union for API error responses**: The server returns `{ error: string, message: string }` but the `error` field values (`'validation'`, `'blocked'`, `'invalid_response'`, `'server_error'`, `'method_not_allowed'`) could be a union type. Low priority — the client doesn't branch on these values.

## 5. Error Handling Infrastructure Assessment

### Current State: Good

- **Custom error class**: `ApiError` exists with `statusCode` field — properly extends `Error`, sets `name`
- **Consistent error format**: All server responses use `{ error: string, message: string }` shape
- **No information leakage**: Server catches all errors and returns themed messages, never raw error details
- **Proper abort handling**: `AbortError` is caught and silently ignored (correct UX behavior)
- **No React error boundary**: Acceptable — the app uses a 4-state machine with an explicit error state, and there are no lazy-loaded components or Suspense boundaries

### What's Good
- Error responses never leak stack traces or internal paths
- Error classification (validation vs server_error) is consistent
- Client-side error handling transitions to error UI state properly
- JSON parsing in Claude API response is already wrapped in try/catch

### What Was Missing (Now Fixed)
- JSON parsing on success-path `response.json()` and `request.json()` had no error handling
