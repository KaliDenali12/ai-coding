## 3. Naming Conventions

### URL Conventions

| Dimension | Convention Used | Consistent? |
|-----------|----------------|-------------|
| Path casing | lowercase (`/api/generate`) | Yes (1/1) |
| Resource vs action | Action-oriented (`generate`) | N/A — single action endpoint, not a CRUD resource |

**Assessment**: The URL `/api/generate` is action-oriented rather than resource-oriented. For an AI generation endpoint that doesn't create a persistent resource, this is acceptable. A REST purist might prefer `POST /api/conspiracies` (creating a conspiracy resource), but since the data is ephemeral and not stored, the current naming is pragmatic and clear.

### Request Field Conventions

### Response Field Conventions

| Dimension | Convention | Examples |
|-----------|-----------|----------|
| Field casing | snake_case | `case_file_number`, `classification_level`, `font_category` |

**FINDING-01: Mixed casing between request and response fields**

- **Request**: camelCase (`conceptA`, `conceptB`)
- **Response**: snake_case (`case_file_number`, `classification_level`, `font_category`)
- **Severity**: Low
- **Impact**: Inconsistency that developers must remember. Not a bug.
- **Recommendation**: Pick one convention. snake_case is dominant (response has more fields). However, changing either would be a breaking change for the client. Document as the chosen convention and move on.
- **Action**: Document only — changing field names is a breaking change.

**FINDING-02: POST success returns 200 instead of 201**

- **Current**: `POST /api/generate` returns `200` on success
- **HTTP convention**: POST that creates a resource should return `201 Created`
- **Severity**: Very Low
- **Assessment**: This endpoint doesn't create a persistent resource — it generates ephemeral content. `200` is actually more semantically accurate than `201` here, since nothing is "created" in the storage sense. The response is a computation result, not a new resource with a URI.
- **Verdict**: Current behavior is **correct** for this use case. No change needed.

- No idempotency key needed — results are ephemeral entertainment, not financial transactions

## 5. Error Response Consistency

### Error Response Shapes

All error responses follow a consistent pattern:

```json
{ "error": "<error_type>", "message": "<user-facing message>" }
```

**Except one:**

| Status | Current Shape | Consistent? |
|--------|--------------|-------------|
| 400 (validation) | `{ error: "validation", message: "..." }` | Yes |
| 400 (blocked) | `{ error: "blocked", message: "..." }` | Yes |
| 405 | `{ error: "Method not allowed" }` | **No** — missing `message` field |
| 500 | `{ error: "server_error", message: "..." }` | Yes |
| 502 | `{ error: "invalid_response", message: "..." }` | Yes |

**FINDING-03: 405 response missing `message` field**

- **Current**: `{ error: "Method not allowed" }` — the `error` field contains a human-readable string instead of a machine-readable code, and `message` is missing
- **Expected**: `{ error: "method_not_allowed", message: "Method not allowed" }`
- **Severity**: Low
- **Impact**: Client error handling expects `message` field (see `api.ts:68-71`). A 405 response would fall back to "Request failed" instead of showing the actual error. In practice, the client only sends POST, so this path is never hit from the app.
- **Action**: **Fixed** — changed to `{ error: "method_not_allowed", message: "Method not allowed" }`. Test updated.

### Error Quality Assessment

| Criterion | Assessment |
|-----------|-----------|
| Messages helpful and specific? | Yes — validation errors specify the problem; blocked/server errors use themed messages |
| All field errors returned at once? | No — fail-on-first for validation (e.g., missing conceptA stops before checking conceptB) |
| Machine-readable error codes? | Yes — `error` field contains coded types (`validation`, `blocked`, `server_error`, `invalid_response`) |
| Sensitive info leaked? | No — server errors return a themed message, never raw errors. `console.error` logs internally only. |

**FINDING-04: Validation fails on first error**

- **Current**: If `conceptA` is missing, the endpoint returns immediately without checking `conceptB`
- **Best practice**: Return all validation errors at once so the user can fix everything in one round-trip
- **Severity**: Very Low
- **Impact**: The client validates inputs before sending, so multi-error responses would rarely be seen. The current behavior is simpler and acceptable for a single-endpoint entertainment app.
- **Action**: Document only — not worth the complexity for this use case.

### Validation Behavior

| Criterion | Assessment |
|-----------|-----------|
| Every input-accepting endpoint validated? | Yes — thorough 3-layer validation |
| Consistent library/approach? | Yes — manual validation (no Zod/Joi), but pattern is consistent |
| Where does validation happen? | Handler level (server), component level (client) |
| Consistent failure status code? | Yes — 400 for all input errors |
| Consistent error format? | Yes — `{ error, message }` (except 405, see FINDING-03) |

**FINDING-05: No Content-Type header validation on incoming requests**

- **Current**: The handler calls `request.json()` directly without checking `Content-Type: application/json`
- **Impact**: Requests with wrong content type may fail with unhelpful errors (e.g., `request.json()` throwing on form-encoded data)
- **Severity**: Very Low
- **Assessment**: Netlify Functions handle this at the platform level. The `request.json()` call will throw on non-JSON, which is caught by the try/catch and returns a 500. This could be improved to return a 400 with a helpful message, but it's not a real risk since only the app's own client calls this endpoint.
- **Action**: Document only — platform handles this adequately.

### Duplicate Validation Logic

**FINDING-06: Blocklist and normalization logic duplicated between client and server**

- **Files**: `src/lib/blocklist.ts` and `netlify/functions/generate.ts`
- **Current state**: Both files contain identical `BLOCKED_TERMS`, `SUBSTITUTIONS`, `normalizeInput()`, and `isBlocked()` — maintained in sync manually
- **Risk**: Drift between the two copies could create bypass vectors (client allows something server blocks, or vice versa)
- **Severity**: Medium (documented in CLAUDE.md as known pitfall)
- **Assessment**: This is a deliberate architectural choice — the serverless function must be self-contained (no imports from `src/`). The duplication is noted in CLAUDE.md. The existing test suites cover both copies.
- **Action**: Already documented. Could be mitigated by extracting to a shared `lib/` at build time, but this adds build complexity. Current approach with tests is acceptable.

## 8. Miscellaneous API Quality

### Rate Limiting

**FINDING-07: No rate limiting on the generate endpoint**

- **Current**: No rate limiting at the application level
- **Platform-level**: Netlify Functions have platform-level rate limits (varies by plan)
- **Risk**: Each request costs money (Claude API call). Without rate limiting, a malicious actor could run up the API bill by spamming requests.
- **Severity**: Medium
- **Assessment**: For a personal/demo project, Netlify's platform limits may suffice. For production with real billing concerns, application-level rate limiting (or Netlify's built-in rate limiting features) should be added.
- **Action**: Recommend adding rate limiting if deploying to production with a funded API key.

### Versioning

- No API versioning. Acceptable — single endpoint, no external consumers, no backward compatibility concerns.

### Content Types

- **Response**: All responses include `Content-Type: application/json` header. Correct.
- **Request**: Client sends `Content-Type: application/json`. Server doesn't validate it (see FINDING-05).

### CORS

- Not explicitly configured. Netlify's default CORS applies. Since the client and API are on the same origin (same Netlify site), this is correct.

### Caching

- No caching headers. Correct — responses are unique AI-generated content that should never be cached.

### Request Tracing

- No request IDs or correlation headers. Acceptable for a single-endpoint entertainment app.
