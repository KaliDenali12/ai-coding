## 3. High-Priority Bugs

### BUG-H1: PolaroidCard — Framer Motion transform conflict likely drops card rotation

**Location**: `src/components/PolaroidCard.tsx:131-133`
**Confidence**: Medium (70%) — requires visual confirmation
**Status**: Document Only — needs visual testing; no automated test can verify

**Description**: The animated PolaroidCard sets rotation via inline `style={{ transform: \`rotate(${rotation}deg)\` }}` on a `<motion.div>` that also uses `initial`, `animate`, and `whileHover` animation props. Framer Motion manages transforms internally via individual properties (`scale`, `y`, etc.) and composes them into a single CSS `transform`. When the entrance animation completes (`animate={{ opacity: 1, scale: 1, y: 0 }}`), Framer Motion generates a transform like `translateY(0px) scale(1)`, which overrides the inline `style.transform: rotate(Xdeg)`.

**Impact**: All cards may render at 0 degrees rotation after their entrance animation completes, instead of their intended random +-4 degree tilt. The corkboard loses its "messy bulletin board" feel and looks artificially grid-aligned.

**Trigger**: Every board render, after the entrance animation finishes (~0.5s per card).

**Suggested fix**: Use Framer Motion's `rotate` prop instead of `style.transform`:
```tsx
<motion.div
  initial={{ opacity: 0, scale: 0.8, y: -20, rotate: rotation }}
  animate={{ opacity: 1, scale: 1, y: 0, rotate: rotation }}
  whileHover={{ scale: 1.03, zIndex: 50 }}
  className="w-[150px] md:w-[200px]"
>
```

**Status**: **FIXED** — commit `aa1016e`. Used Framer Motion `rotate` prop in `initial`/`animate` instead of `style.transform`. Also added explicit card dimensions (`h-[210px] md:h-[280px]`) to fix BUG-H2 in the same commit.

### BUG-H2: PolaroidCard — CSS card flip container has no explicit height; back face may clip briefing text

**Location**: `src/components/PolaroidCard.tsx:39-106`
**Confidence**: Medium (65%) — complex CSS interaction, needs visual testing
**Status**: Document Only — requires visual confirmation

**Description**: The CSS card flip uses the standard `absolute inset-0` pattern for front/back faces inside a `preserve-3d` container. However, no element in the chain has an explicit height:

1. `motion.div` wrapper: `w-[150px] md:w-[200px]` — width only, no height
2. `.perspective` div: no dimensions (but `perspective: 1000px` creates a containing block)
3. `.preserve-3d` div: `width: 100%; height: 100%` — circular dependency with parent
4. Front/back faces: `absolute inset-0` — sizes to containing block (the `.perspective` div)

The `.perspective` div's height comes from its content, but its only child is the `.preserve-3d` div whose children are all absolute (removed from flow). This means the containing block may have 0 computed height. The front face content overflows visually (default `overflow: visible`), so it appears correct. But the back face has `overflow-y-auto`, which clips content at the computed height — potentially 0px.

**Impact**: Briefing text on the back of flipped cards may be invisible or not scrollable. Users click to "read briefing" but see nothing.

**Trigger**: Flip any card after the reveal animation completes.

**Suggested fix**: Add explicit card dimensions to the container:
```tsx
<div
  className="perspective cursor-pointer"
  style={{ width: cardWidth, height: cardHeight }}
  // ... or use Tailwind: className="perspective cursor-pointer w-[150px] h-[210px] md:w-[200px] md:h-[280px]"
>
```

**Status**: **FIXED** — commit `aa1016e`. Added `h-[210px] md:h-[280px]` to both animated and non-animated card wrappers, giving the CSS flip container explicit dimensions.

## 4. Medium-Priority Bugs

### BUG-M1: seededRandom returns inclusive [0, 1] instead of [0, 1)

**Location**: `src/lib/layout.ts:21`
**Confidence**: High (95%)
**Status**: Document Only — fix changes all board layouts (user-facing behavior change)

**Description**: The seeded random number generator divides by `0xffffffff` (4,294,967,295):
```ts
return (s >>> 0) / 0xffffffff
```
This produces a range of [0.0, 1.0] inclusive. Standard RNG convention (and `Math.random()` spec) is [0, 1) exclusive of 1. The correct divisor is `0x100000000` (4,294,967,296 = 2^32).

When `random()` returns exactly 1.0:
- `(1.0 - 0.5) * 8 = 4.0` rotation (normally max is ~3.999)
- `(1.0 - 0.5) * usableWidth * 0.4` = slightly more jitter than expected
- All values are clamped by `Math.max/Math.min`, so no out-of-bounds

**Impact**: Negligible. Rotation is 4.0 instead of 3.999 at most. Positions are clamped. This is mathematically incorrect but functionally harmless.

**Trigger**: When the LCG's internal state reaches `0xffffffff` (happens for specific seeds).

**Why not fixed**: Changing the divisor would change all seeded random outputs, altering every board layout. This is a user-facing behavior change (criterion 4 fails). The mathematical impact is negligible.

### BUG-M2: Blocklist bypass via space insertion in single-word terms

**Location**: `src/lib/blocklist.ts:66-68` and `netlify/functions/generate.ts:52-53`
**Confidence**: High (90%)
**Status**: Document Only — design tradeoff, requires product decision

**Description**: `normalizeInput()` collapses whitespace but preserves spaces (necessary for multi-word term matching like "school shooting"). This means single-word blocked terms can be bypassed by inserting spaces:

- Input `"nig ger"` normalizes to `"nig ger"` (space preserved)
- `includes("nigger")` returns false — bypass successful
- But `"school shooting"` correctly normalizes to `"school shooting"` and matches

The separator strip only removes `[_.+-]+`, not spaces.

**Impact**: Determined users can bypass single-word slur filters. Server-side Claude system prompt (Layer 3) would still catch most cases, but the blocklist layer is defeated.

**Trigger**: Any single-word blocked term with a space inserted (e.g., "h itler", "fa ggot").

**Suggested fix**: Run the blocked term check twice — once with spaces and once without:
```ts
const withoutSpaces = normalized.replace(/\s+/g, '')
return BLOCKED_TERMS.some((term) => {
  const termNoSpaces = term.replace(/\s+/g, '')
  return normalized.includes(term) || withoutSpaces.includes(termNoSpaces)
})
```

**Status**: **FIXED** — commit `528875f`. Added dual-pass check: `isBlocked()` now checks both with spaces (for multi-word terms) and without spaces (catches single-word bypass). Updated in both `blocklist.ts` and `generate.ts`. Added 3 new tests.

### BUG-L2: ApiError status code is misleading on successful-response JSON parse failure

**Location**: `src/lib/api.ts:98`
**Confidence**: High (90%)
**Status**: Document Only — low impact, only logged

**Description**: When `response.ok` is true but `response.json()` throws (malformed JSON from server), the ApiError is constructed with `response.status` (a 2xx code):
```ts
throw new ApiError('Our server returned an unreadable response. Please try again.', response.status)
```

An error with a 200 status code is semantically confusing. The status code is only used in logging, so the practical impact is minimal, but it could mislead debugging.

**Suggested fix**: Use a fixed error status like 502:
```ts
throw new ApiError('Our server returned an unreadable response. Please try again.', 502)
```

**Status**: **FIXED** — commit `9781577`. Changed `response.status` to hardcoded `502`.

**Analysis**: State machine is clean. No impossible transitions, no stuck states, no missing guards. The `AbortError` silent handling is correct — it prevents the error screen from showing when a request is intentionally cancelled.

**Race condition check**: If `handleTimeout` fires (sets screen to `error` + aborts fetch) while the fetch is resolving, the abort triggers `AbortError` in the catch block, which returns silently. The screen stays at `error`. Correct.

## 11. Recommendations

### Immediate (verify visually)
1. **Test PolaroidCard in a browser**: Confirm whether cards have rotation tilt and whether flipped cards show briefing text. If either fails, BUG-H1/H2 are confirmed critical.

### Short-term
2. **Add explicit height to card flip container**: Even if it works now, the CSS card flip pattern without explicit dimensions is fragile. Adding `h-[210px] md:h-[280px]` to the perspective container makes it robust.
3. **Use Framer Motion `rotate` prop**: Replace `style={{ transform: 'rotate(...)' }}` with `rotate` in animation props for guaranteed compatibility.
4. **Add blocklist test for example pairs**: `EXAMPLE_PAIRS.forEach(pair => expect(checkInputs(pair.a, pair.b).valid).toBe(true))`

### Medium-term
5. **Consider dual-pass blocklist check**: Run `includes()` both with and without spaces to resist space-insertion bypasses while preserving multi-word matching.
6. **Fix seededRandom divisor**: Change `0xffffffff` to `0x100000000` in a release where layout changes are acceptable (or make the change alongside a deliberate layout refresh).
